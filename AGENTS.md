# AI 办事项目协作原则

## 产品定位

AI 办事不是把学校已有办事入口简单换成聊天框，而是压缩师生从“有需求”到“完成办理”的时间。

过去流程是：师生有需求 -> 登录数智安农 -> 进入安心办 -> 浏览或搜索事项 -> 判断哪个能办 -> 手动填写信息 -> 提交。

我们的目标是：师生直接说需求，系统理解意图、匹配事项、带入可信画像和业务数据，尽量减少查找、判断和重复填写。

## 已经形成的系统能力

### 1. 智能办事导航

- 手机端 H5 是主入口，访问 `https://xgaigc.ahau.edu.cn`。
- 首页进入后通过学校 OAuth 获取用户信息，并写入 `user_profiles`。
- AI 对话入口支持自然语言提问，例如“云盘怎么用”“学生档案去哪查”“体育场馆预约”。
- 回答优先来自数据库中的结构化事项，不让大模型编造办理入口。
- 返回内容以办事卡片为主，包括入口、适用对象、办理机构、联系人、电话、办理时间、流程、附件、二维码等。
- “去办理”按钮必须跳转数据库中的可信链接。

### 2. OAuth 和用户画像

- OAuth 回调地址使用 `https://xgaigc.ahau.edu.cn/callback`。
- OAuth 原始数据写入 `oauth_raw`，规范化信息写入用户画像字段。
- 用户画像字段包括姓名、学号/工号、身份、学院、专业、班级、手机号、邮箱等。
- 性别字段以 OAuth 原始数据为准，不再做反转处理。
- 后续画像扩展进入 `user_memories` 和 `user_preference_profiles`，用于记忆、偏好和推荐。

### 3. 事项库

统一事项库仍以 `service_items` 为核心表。

已支持字段包括：

- 事项名称、分类、描述、办理入口。
- 面向对象、办理机构、联系人、联系电话、办理时间。
- 办理依据、申请材料、办理流程、注意事项、常见问题。
- 关键词、来源链接、状态、核验时间。

附件、二维码、流程图等统一进入 `service_item_assets`。

事项状态规则：

- `ENABLED`：可搜索、可推荐、可回答。
- `DISABLED`：已知不可用、入口异常、暂不推出，用户问到时不直接给入口。
- `DRAFT`：未确认，默认不作为正式结果。

### 4. 身份过滤

系统按学校给的“面向对象”做身份过滤。

统一身份类型先按五类理解：

- 教职工
- 本科生
- 研究生
- 校友
- 访客（社会人员）

兼容规则：

- “全部人员”表示上述身份均可使用。
- “研究生校友”至少允许研究生、校友和研究生校友相关用户命中。
- “资环学院教职工”等学院限定表达，当前在无法准确判断学院时，先允许教职工命中，但后续应支持学院精细过滤。
- 学校原始“面向对象”原则上原样搬运，不主观改窄。

### 5. AI 分工

数据库负责：

- 确定办理入口。
- 确定流程、对象、材料、联系人。
- 确定事项是否可用。

Dify / 大模型负责：

- 理解用户自然语言。
- 辅助解释流程。
- 引导用户补充信息。
- 对无结果问题进行总结和归类。

大模型不能直接编造链接、流程和办理条件。

MiniMax 定位：

- 记忆提取。
- 用户意图归类。
- 未满足需求聚类。
- 引导用户追问或澄清。
- 后台异步总结，不阻塞用户即时问答。

## 闭环思路

### 1. 痛点优先

首页推荐不按“大而全”排序，而按师生真实痛点优先。

优先选择：

- 办理频率高。
- 填错成本高。
- 咨询量高。
- 时效性强。
- 流程复杂。
- 入口容易找不到。

痛点来源包括两类：

- 学生和老师的主动反馈、问卷、访谈、群聊收集。
- AI 办事后台数据，例如搜索最多、点击最多、追问最多、失败最多、跳转最多的业务。

### 2. 自动带入信息

对于高频痛点事项，后续要逐步支持“少填、预填、免填”。

例如交空调费、宿舍相关、证明申请、离校、预约类事项，可由老师或业务系统把可信数据推给我们，系统自动带入姓名、学号、学院、班级、宿舍、联系方式等信息，减少用户填错。

所有自动带入的数据必须来自可信来源：

- OAuth。
- 学校业务库。
- 老师推送数据。
- 已确认画像。

不能由大模型编造。

### 3. 压缩办事时间

AI 办事要优化三个时间：

- 登录时间：尽量复用统一身份认证，减少重复登录。
- 查找时间：用户不用看事项列表，直接问即可。
- 判断时间：系统根据身份、画像、场景、历史行为推荐最相关事项，并说明依据。

### 4. 个性化体验

系统要根据用户身份和行为做个性化推荐。

例如用户经常点击某类业务，下次首页可以优先展示；毕业生、新生、教职工、研究生、本科生看到的推荐应不同。

个性化推荐必须可解释，不能为了“智能”牺牲准确性。

### 5. 监测后台

监测后台用于从时间、身份、业务类型等角度评估 AI 办事效果。

当前已经支持或正在形成的指标：

- 总访客量和今日活跃。
- 30 天访问人数趋势。
- 身份占比。
- 学生常问问题。
- 老师常问问题。
- 事项点击排行。
- 未满足需求池。
- 人工处理未满足需求。

后台原则：

- 手机端优先。
- 图表优先，少堆列表。
- 只展示当前最有决策价值的数据。
- 不做没有数据支撑的“年度趋势”等伪图表。
- 后台数据要反向指导首页痛点推荐和事项库维护。

## 未满足需求闭环

当用户提问没有命中稳定可办理事项时，系统会记录到 `assistant_turns`。

后台将这些问题聚合成“未满足需求池”，用于发现：

- 事项库缺失。
- 关键词缺失。
- 身份规则过窄。
- 入口暂不可用。
- 用户高频真实需求。

人工处理逻辑：

- 管理员可以给聚类后的需求设置高、中、低优先级。
- 管理员可以隐藏暂不处理的需求。
- 管理员可以标记“已落实并通知”。
- 已落实后，系统给问过该问题的用户生成站内提醒，提醒其回来查看。

相关表：

- `unmet_need_reviews`
- `user_notifications`

## TableX 填报类事项的暂定设计

从 `data1.ahau.edu.cn/TableX/...` 来的新事项属于“填报类事项”，和普通办事指南不同。

这类事项通常包含：

- 填报开始时间。
- 填报截止时间。
- 填报分类。
- 填报状态。
- appId。
- taskId。
- appName。
- 二维码。
- 填报链接。

暂定原则：

- 不另起一套事项系统。
- 主数据仍然进入 `service_items`，保证 AI 搜索、身份过滤、办事卡片继续统一。
- 二维码继续进入 `service_item_assets`，`asset_type = 'qrcode'`。
- 如老师确认需要长期维护填报状态，则新增 `service_item_form_meta` 子表保存 TableX 扩展字段。

待向老师确认的问题：

- 这些 TableX 填报应用是否属于正式“办事事项”，需要进入 AI 办事统一搜索？
- 已过期事项是否要入库但设为 `DISABLED`，还是不入库？
- “已填报”状态是面向当前登录用户，还是事项整体状态？
- 二维码是否需要展示给用户，还是只展示“去填报”链接？
- 填报开始/截止时间是否需要在 AI 回答中明确提示？
- 是否能提供批量导出字段，而不是人工从页面逐项复制？
- 是否后续需要根据用户身份判断“该用户是否已填报”？

如果老师确认这批数据后续会持续同步，建议新增表：

```sql
CREATE TABLE IF NOT EXISTS `service_item_form_meta` (
  `id` VARCHAR(191) NOT NULL,
  `service_item_id` VARCHAR(191) NOT NULL,
  `platform` VARCHAR(50) NOT NULL DEFAULT 'TableX',
  `app_id` VARCHAR(100) NULL,
  `task_id` VARCHAR(191) NULL,
  `app_name` VARCHAR(255) NULL,
  `fill_type` VARCHAR(50) NULL,
  `flag` VARCHAR(50) NULL,
  `form_type` VARCHAR(50) NULL,
  `fill_category` VARCHAR(100) NULL,
  `fill_status` VARCHAR(50) NULL,
  `start_at` DATETIME(3) NULL,
  `end_at` DATETIME(3) NULL,
  `raw_url` TEXT NULL,
  `raw_data` JSON NULL,
  `created_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  `updated_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
  PRIMARY KEY (`id`),
  UNIQUE KEY `service_item_form_meta_service_item_id_key` (`service_item_id`),
  KEY `service_item_form_meta_platform_idx` (`platform`),
  KEY `service_item_form_meta_fill_status_idx` (`fill_status`),
  CONSTRAINT `service_item_form_meta_service_item_id_fkey`
    FOREIGN KEY (`service_item_id`) REFERENCES `service_items` (`id`)
    ON DELETE CASCADE ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

## 数据表当前职责

- `user_profiles`：OAuth 用户主画像。
- `user_preference_profiles`：用户偏好画像。
- `user_memories`：可积累的用户记忆。
- `service_items`：统一事项库。
- `service_item_assets`：事项图片、二维码、流程图、附件入口。
- `user_events`：用户点击、跳转、行为事件。
- `assistant_turns`：用户提问和 AI 回答记录。
- `recommend_rules`：首页推荐规则。
- `unmet_need_reviews`：未满足需求人工处理状态。
- `user_notifications`：站内提醒。
- `service_item_form_meta`：暂定，TableX 填报类事项扩展信息，待老师确认。

## 实施顺序

### 第一阶段：智能办事导航

- 完成 OAuth 登录和用户画像入库。
- 建立结构化办事事项库。
- 支持 AI 问答、事项卡片、办理入口、附件和相关链接。
- 按学校给定“面向对象”做身份过滤。
- 首页推荐先人工配置高频痛点事项。

### 第二阶段：痛点事项优化

- 选择少量最痛事项深做，例如缴费、宿舍、证明、离校、预约类业务。
- 接入老师推送或业务库数据。
- 做表单预填和信息校验。
- 减少重复输入和填错。

### 已确定的第一个痛点事项：电费转账

电费转账的核心痛点是学生容易选错楼宇、楼层、房间，导致充错房间。

数据来源：

- 表名：`xszsxx`
- 所在库：`aibs`
- 字段：`XH`, `XM`, `LD`, `FJ`, `CW`
- `XH` 对应 OAuth 返回的用户 `id`

字段理解：

- `XH`：学号。
- `XM`：姓名。
- `LD`：楼栋编码。
- `FJ`：房间号。
- `CW`：床位号，仅用于核对。

区域规则：

- `LD` 以 `N` 开头：选择 `校本部宿舍`。
- `LD` 以 `B` 开头：选择 `大学生公寓`。

楼宇规则：

- `N12` 表示校本部宿舍 `12栋`。
- `B01` 表示大学生公寓 `1栋`。
- 楼栋数字取首字母后的数字并去掉前导 0。

页面楼宇选项规则：

- 校本部照明：`校本部宿舍·照明_X栋`。
- 校本部空调：`校本部宿舍·空调_X栋`。
- 大学生公寓照明：`大学生公寓·照明_X栋`。
- 大学生公寓空调：`大学生公寓·空调_X栋`。

楼层规则：

- `FJ = 711` 表示第 `7` 层。
- `FJ = 415` 表示第 `4` 层。
- 若未来出现四位及以上房间号，例如 `1012`，楼层按去掉最后两位后的数字理解，即第 `10` 层。

房间显示规则：

- 校本部宿舍显示为：`12栋711室`。
- 大学生公寓显示为：`K1-415`。

回答策略：

- 用户明确问“空调电费/空调费”时，重点给空调方案。
- 用户明确问“照明电费/照明费”时，重点给照明方案。
- 用户只说“电费/充电费/电费转账”时，同时给照明和空调两个方案。
- 查询不到住宿信息、`LD = 走读`、`FJ` 为空时，不猜测，提醒用户先联系辅导员或宿管确认。
- 回答中必须提醒“付款前核对楼宇、楼层、房间，支付后持卡前往自助多媒体写卡完成余额更新”。

### 第三阶段：个性化推荐

- 根据用户身份、时间节点、点击行为、常问事项做首页推荐。
- 支持“常用事项”“最近办理”“你可能要办”。
- 保持推荐结果可解释。

### 第四阶段：监测后台

- 建立数据看板。
- 做环比、类比、趋势分析。
- 根据数据持续优化事项库、推荐位和 AI 回答。

## 决策原则

- 准确性优先于炫技。
- 手机端体验优先。
- 入口和流程必须来自数据库或学校可信来源。
- 大模型负责理解、解释和辅助表达，不直接编造办理链接。
- 学校给的事项信息原则上原样搬运，不主观改窄“面向对象”。
- 高频痛点事项值得深做，低频事项先保证可查、可点、可追溯。
- 没有稳定数据来源的图表先不做，避免后台看起来“很炫但不可信”。
- 新类型数据先兼容统一事项库，只有确实有长期扩展字段时才加子表。
